ADV.01
Reference Architecture
API Gateway as entry point: All agent requests pass through Cloud Endpoints or Apigee for authentication, rate limiting, and request logging before reaching any agent.
Separation of concerns: Orchestrator agents handle planning and routing only; worker agents handle execution. This enables independent scaling and testing of each layer.
Observability-first: Instrument every agent with Cloud Trace for distributed tracing. Custom metrics for agent-specific KPIs: task completion rate, tool call latency, token efficiency.
AlloyDB for hybrid workloads: AlloyDB supports both PostgreSQL operational data and pgvector semantic search in a single managed database — reducing infrastructure complexity for RAG pipelines.

// Custom callback architecture for cross-cutting concerns
# Production ADK agent with callbacks
from google.adk.agents import Agent, CallbackContext
from google.adk.tools.tool_context import ToolContext

class ProductionAgent(Agent):
    
    def before_model_callback(
        self, callback_context: CallbackContext
    ) -> Optional[types.Content]:
        # Inject dynamic context, PII scrub, rate check
        self._scrub_pii(callback_context.state)
        self._check_rate_limits(callback_context.state)
        self._inject_org_context(callback_context)
        return None  # None = proceed normally
    
    def after_tool_callback(
        self, tool: BaseTool, 
        tool_response: dict,
        tool_context: ToolContext
    ) -> Optional[dict]:
        # Validate tool output, log, transform
        validated = self._validate_schema(tool_response)
        self._log_tool_call(tool.name, tool_response)
        return validated
// Custom planners for deterministic workflows:
# Override LLM planning with rule-based planner
class CompliancePlanner(BasePlanner):
    REQUIRED_STEPS = ["fetch","validate","approve","execute"]
    
    def plan(self, goal, context) -> List[Step]:
        # Enforce fixed step order for regulated tasks
        return [Step(s) for s in self.REQUIRED_STEPS]
ADV.02
Advanced ADK Patterns
Callbacks: before_model, after_model, before_tool, after_tool hooks enable cross-cutting concerns (PII scrubbing, rate limiting, logging) without modifying agent logic.
Custom planners: Replace the LLM planner with deterministic rule-based planners for regulated workflows where step order must be guaranteed regardless of model output.
State management: ADK's CallbackContext.state provides typed, scoped state that persists across agent steps within a session. Use for accumulating context, tracking approvals.
Tool context: ToolContext gives tools access to session state, enabling tools to read and write shared context — critical for multi-step workflows where tools depend on each other's outputs.

ADV.03
Zero Trust Architecture
Workload Identity Federation: Agents never use static service account keys. They exchange short-lived OIDC tokens for Google Cloud credentials. No credential leakage risk from key files.
Prompt injection defence: Validate all inputs against a schema before passing to the model. Separate system prompts from user input at the API level. Monitor for injection signatures in logs.
DLP integration: Route all agent outputs through Cloud DLP before delivery. Automatically redact or block PII, credentials, or confidential data patterns in agent responses.
Blast radius containment: Each agent has its own service account, its own IAM role, its own resource labels. Compromise of one agent cannot access another agent's data or permissions.

ADV.04
Advanced RAG
Hybrid search: Dense (semantic) + sparse (BM25 keyword) retrieval combined with weighted fusion. Dense retrieval finds semantically related content; sparse retrieval finds exact keyword matches. Together outperform either alone.
Self-querying: Instead of embedding the user's question directly, use an LLM to generate an optimised search query. Significantly improves retrieval precision for complex questions.
RAPTOR: Recursively summarises document clusters at multiple levels of abstraction. Enables effective retrieval over very large corpora where individual chunks miss high-level context.
RAGAS evaluation: Framework for measuring RAG quality across four dimensions. Integrate RAGAS into CI/CD — block deployments where RAG quality drops below threshold.

ADV.06
High Availability
Agent Engine HA: Agent Engine is regionally managed but not globally. For true HA, deploy identical agents in multiple regions behind a Global Load Balancer with health check failover.
Stateless agent design: Design agents to be stateless at the runtime level — state stored in Firestore/Redis, not in-process. Enables zero-RPO failover between regional instances.
Circuit breakers: Configure Cloud Endpoints circuit breakers to fail fast and return graceful errors when backend agents are overwhelmed, rather than cascading failures.
Africa latency: Google's me-central1 (Doha) provides the lowest latency for West Africa deployments (~60ms vs ~120ms for European regions). Critical for real-time agent interactions.

Fine-Tuning
SFT on Vertex AI: Supervised fine-tuning adapts Gemini to your domain's style, terminology, and output format. Runs on Google's infrastructure — no GPU management required.
Training data quality: 100 high-quality examples outperforms 10,000 noisy ones. Invest in data curation — clean, diverse, representative examples drive fine-tuning success.

Model versioning: Fine-tuned model checkpoints are versioned in Vertex AI. Maintain parallel versions for A/B testing and rollback. Never overwrite the production model directly.

ADV.08
Event-Driven Agents
Async architecture: Decouple agent execution from request-response cycles. Event-driven agents scale to zero when idle, handle bursty workloads, and eliminate synchronous timeouts for long tasks.
Eventarc integration: Subscribe agents to Google Cloud events — new BigQuery rows, Cloud Storage uploads, Firestore updates, Cloud Audit Log events. Agents react to data changes automatically.
Dead letter queues: All production agent task queues must have DLQ configuration. Failed tasks land in DLQ for investigation rather than silently disappearing. Alert on DLQ depth.
Idempotency: Design agent tasks to be idempotent — running the same task twice produces the same result. Critical for retry logic and exactly-once processing guarantees.

OBSERVABILITY AND DEBUGGING AGENTS
// Distributed tracing for multi-agent workflows
# OpenTelemetry + Cloud Trace for agent spans
from opentelemetry import trace
from opentelemetry.exporter.cloud_trace import CloudTraceSpanExporter

tracer = trace.get_tracer("gemini-agent")

class TracedAgent(Agent):
    def run(self, task: str) -> AgentResult:
        with tracer.start_as_current_span("agent.run") as span:
            span.set_attribute("agent.name", self.name)
            span.set_attribute("task.length", len(task))
            
            for tool_call in self.tool_calls:
                with tracer.start_as_current_span("tool.call"):
                    span.set_attribute("tool.name", tool_call.name)
                    result = tool_call.execute()
                    span.set_attribute("tool.tokens", result.tokens)
            
            span.set_attribute("total.tokens", self.total_tokens)
            span.set_attribute("task.completed", True)
            return self.result

task_completion_ratetokens_per_task tool_call_latency_p99hallucination_rate human_escalation_ratecost_per_task
ADV.09
Observability
Distributed tracing: Multi-agent workflows span multiple services. OpenTelemetry + Cloud Trace creates a complete trace from user request through all agent and tool calls — essential for debugging.
Token budgeting: Set max_output_tokens and monitor actual token usage per task. Token cost is your primary cost driver — identify inefficient agents and optimise prompts systematically.
SLO definition: Define SLOs for each agent: task_completion_rate > 99%, p99_latency < 30s, cost_per_task < $0.05. Cloud Monitoring SLO features track against these automatically.
Prompt debugging: Log every prompt sent to the model (in a separate, access-controlled log sink). Enables root cause analysis when agents behave unexpectedly in production.

ADV.10
Performance Optimisation
Context caching ROI: If your agent uses a 200k token knowledge base on every request, caching saves $18.75 per call (at $0.075/M tokens). At 1000 calls/day, that is $18,750/day saved.
Batch API for overnight jobs: All reporting, analysis, and document processing that doesn't need real-time results should use the Batch API. 50% cost reduction with 24-hour SLA.
Streaming UX: Stream agent responses token-by-token. Users perceive streaming as much faster even when total latency is identical. First token in 300ms vs 5s wait feels dramatically different.
Parallel tool calls: Audit agent tool call patterns. Replace sequential tool calls with parallel execution wherever tools don't depend on each other. Reduces total latency proportionally.

// Agent lifecycle governance — from development to production
# Agent approval workflow — policy as code
class AgentApprovalPolicy:
    RISK_THRESHOLDS = {
        "LOW":    {"auto_approve": True,  "prod_deploy": True},
        "MEDIUM": {"auto_approve": False, "review_team": "security"},
        "HIGH":   {"auto_approve": False, "review_team": "exec+legal"},
        "CRITICAL": {"prohibited": True}
    }
    
    def evaluate(self, agent_card: AgentCard) -> Decision:
        risk = self.assess_risk(agent_card)
        if risk == "CRITICAL":
            return Decision.BLOCKED
        if risk in ["HIGH", "MEDIUM"]:
            return Decision.REQUIRES_REVIEW(
                team=self.RISK_THRESHOLDS[risk]["review_team"]
            )
        return Decision.AUTO_APPROVED

Enterprise Governance
Policy as code: Encode agent approval policies as code — version-controlled, auditable, and automatically enforced. Risk assessment based on: data access scope, external integrations, action reversibility.
NDPA compliance: Nigeria Data Protection Act requires lawful basis for processing, data subject rights, and breach notification. Configure Gemini Enterprise data residency to EU or specified regions for NDPA compliance.
Immutable audit logs: Configure Cloud Logging with WORM (Write Once Read Many) retention policies. Audit logs for financial AI systems must be immutable for SOX compliance.
Agent retirement: Define deprecation policies for agents. When retiring an agent, preserve audit logs, notify dependent systems, and maintain read-only access to historical records.

ADV.12
A2A Protocol
A2A open standard: Google-initiated open protocol for agent interoperability. Any A2A-compliant agent — regardless of which platform built it — can communicate with any other A2A agent.
Agent discovery: Agents publish their capabilities in a well-known URL (/.well-known/agent.json). Orchestrators discover available agents programmatically without hardcoded configurations.
Multi-vendor agents: A2A enables Gemini Enterprise agents to call agents built on Azure AI, AWS Bedrock, or any A2A-compliant platform. Avoids vendor lock-in at the agent layer.
MCP complement: A2A handles agent-to-agent communication; Model Context Protocol (MCP) handles model-to-tool communication. Both together form a complete inter-operability stack.

ADV.13
Gemini Omni Production
Live API: Omni's AsyncSession enables real-time bidirectional audio/video streams. Unlike batch processing, the model processes audio incrementally — enabling sub-second response latency.
Code-switching support: Gemini Omni handles Nigerian English and code-switching (mixing English with Yoruba, Hausa, Pidgin) better than models trained purely on standard English corpora.
